## 一、搭建思路

### 1.1 痛点识别：渠道销售日常运营中的三个「隐形消耗」

在搭建这个 Skill 之前，我对自己和团队过去一年的日常工作进行了梳理，发现有三个环节构成了显著的「隐形消耗」：

| 痛点 | 具体表现 | 日均耗时 | 核心问题 |
|------|---------|---------|---------|
| **内容搜集** | 每天刷 36kr、牛透社、飞书更新日志、开放平台公告，筛选哪些新闻和功能更新值得同步给伙伴 | 30-45 分钟 | 人工搜索、碎片化、容易遗漏 |
| **文案整理** | 找到内容后要自己精简、改写，适配伙伴阅读习惯（不能直接转发原文） | 20-30 分钟 | 重复劳动、格式不统一、质量不稳定 |
| **定向分发** | 不同能力的伙伴需要不同的内容——新伙伴要基础培训、核心伙伴要竞品分析、低效伙伴要销售技巧，但手动区分太麻烦，往往一刀切群发 | 15-20 分钟 | 分组粗放、内容错配、无法追踪 |

三项合计每天至少消耗 **1 小时以上**，本质上是把渠道销售经理的时间从「高价值的伙伴关系管理和业务增长」拉回到了「信息搬运工」的角色上。

### 1.2 业务洞察：渠道伙伴赋能不能止于「发通知」

深入看，这个问题的本质不是「缺工具」，而是缺一套**系统化的内容运营机制**：

- **信息时效性 = 渠道信任度。** 伙伴对飞书的感知很大程度取决于渠道经理传递信息的及时性和专业度。如果我们每次都是过了好几天才转二手信息，伙伴会觉得我们不专业。
- **内容差异化 = 赋能精准度。** 一刀切群发是最低效的运营方式。新签约伙伴还在学基础功能，你给他发竞品分析他看不懂；核心伙伴天天在打大客户，你给他发入门 FAQ 是浪费时间。
- **可回溯 = 持续改进的基础。** 过去发完就完了，不知道发了多少次、哪些发了哪些没发、哪些伙伴接收了什么内容。没有数据就无法复盘和改进。

这个 Skill 的设计核心就是：**把「信息搜集→加工→分发→归档」这个链条自动化，让渠道销售经理从执行者变成决策者**——你决定推什么、推给谁，AI 帮你完成剩下的全部操作。

### 1.3 技术选型：为什么选择 lark-cli + AI Agent 而非自建系统

| 方案 | 开发成本 | 维护成本 | 与飞书集成度 | 灵活性 |
|------|---------|---------|------------|-------|
| 自建 Web 后台 | 高（前后端 + 数据库 + 定时任务） | 高 | 需逐个对接飞书 API | 低 |
| 飞书多维表格 + 工作流 | 中 | 中 | 原生集成 | 中 |
| **lark-cli Skill + AI Agent** ✅ | **低（一份 SKILL.md）** | **极低（Agent 自动执行）** | **原生集成（wiki/im/docs/base）** | **极高** |

选择 AI Agent 路线的关键考量：
- **飞书 CLI 已开源且获 12.2k Stars**，覆盖 12 个核心业务能力，底层轮子不需要自己造
- **Skill 本质是一份 Markdown 文件**，可版本管理、可 Git 协作、可随时修改生效
- **AI Agent 能理解自然语言指令**，不需要菜单/按钮/表单，销售经理用说话的方式就能操作
- **WebSearch + WebFetch 天然具备实时性**，AI Agent 可以直接搜索互联网获取最新资讯，无需维护 RSS 订阅或爬虫

### 1.4 架构设计：四大模块 + 双模式

```
用户自然语言指令
        │
        ▼
┌─────────────────────────────────────────────────┐
│                  AI Agent                        │
│                                                  │
│  模块0: 自动实时采集    ← WebSearch + WebFetch    │
│  模块1: 内容归集与分类  ← lark-wiki（知识库）     │
│  模块2: 推送管理        ← lark-im（群聊消息）     │
│  模块3: 知识库查询复用  ← lark-docs（全文搜索）   │
│                                                  │
│  双模式：自动采集 + 手动录入                      │
└─────────────────────────────────────────────────┘
```

**模块0 是核心差异化能力。** 区别于简单的「转发助手」，这个 Skill 会主动执行多轮 WebSearch（综合搜索→垂直媒体深度→竞品动态），自动筛选 48 小时内的高质量内容，标注来源 URL 和发布时间，生成预览供审核——确保每一条推送给伙伴的信息都是实时、可溯、经过人工把关的。

## 二、使用流程

### 2.1 每日标准流程（3 步，总计约 10 分钟人工时间）

```
09:30  系统自动采集 → 输出预览
         ↓
10:00  你花 3-5 分钟确认/删改 → 归档至知识库
         ↓
16:00  生成推送文案 → 你确认后一键发送至全体/指定分组
         ↓
       （推送前的确认环节，按需可选定向推送）
```

### 2.2 核心操作指令速览

| 场景 | 用户说什么 | Agent 做什么 |
|------|-----------|------------|
| **每日采集** | 「采集今日全部内容」 | 并行 WebSearch（行业+飞书+竞品）→ 输出预览 |
| **补充搜索** | 「再搜一下政企数字化政策」 | 定向补充搜索 → 追加到预览 |
| **手动录入** | 「上传培训资料：丢单应对技巧...」 | 创建 wiki 节点 → 自动分类 → 生成推送文案 |
| **每日推送** | 「生成今日推送」 | 聚合当日归档内容 → 格式化为精简文案 → 预览 → 确认后发送 |
| **定向推送** | 「把培训资料推送给低效伙伴」 | 检索内容 → 匹配分组 → 附言差异化 → 发送 |
| **紧急通知** | 「生成加急推送：定价调整」 | 创建加急节点 → 关键变更+行动项 → 全体+核心 |
| **知识库查询** | 「查近7天所有培训资料」 | 知识库检索 → 表格输出 |
| **学习追踪** | 「查看低效伙伴本月接收内容」 | 推送历史检索 → 覆盖评估 |
| **周度复盘** | 「查看本周推送清单」 | 推送覆盖统计 → 分组缺失提醒 |

### 2.3 核心交互理念

整个流程的设计遵循一个原则：**AI 负责「找」和「写」，人负责「选」和「发」。**

- Agent 永远不会在未经你确认的情况下直接推送
- 采集预览只呈现客观信息（标题+摘要+来源+链接），不做主观推荐
- 你告诉 Agent「推 1、3、5」，它就精确执行，不多推不漏推

## 三、应用价值

### 3.1 效率提升

| 指标 | 使用前（人工） | 使用后（Skill） | 节省 |
|------|-------------|---------------|------|
| 每日内容搜集 | 30-45 分钟 | 0（自动） | 100% |
| 推送文案整理 | 20-30 分钟 | 3-5 分钟（审核确认） | ~85% |
| 分组定向分发 | 15-20 分钟 | 1-2 分钟（一句话指令） | ~90% |
| 知识库查询 | 5-10 分钟（翻聊天记录） | 30 秒（关键词一句） | ~90% |
| **每日合计** | **~80 分钟** | **~10 分钟** | **~87%** |

按每位渠道销售经理计算，每天省下约 70 分钟，每月约 23 小时——相当于每个月多出近 3 个工作日用于拜访伙伴、推进商机。

### 3.2 能力升级：从「信息搬运」到「精准运营」

| 维度 | 使用前 | 使用后 |
|------|-------|-------|
| **信息时效** | 依赖个人刷新闻习惯，常有遗漏 | Agent 多轮搜索+48h 过滤，每条标注来源和发布时间 |
| **内容质量** | 直接转发原文链接，伙伴阅读率低 | 自动提炼为≤3行精简摘要+口语化改写，适配销售阅读场景 |
| **分组精准度** | 全群群发为主，偶尔手动区分 | 四级分组（全体/核心/新签约/低效）+ 差异化附言 |
| **知识沉淀** | 聊天记录刷没了就找不到 | 全部归档至飞书知识库，支持标签/日期/关键词检索 |
| **可追溯性** | 不知道发了什么、发给谁 | 每推送记录归档，可查询伙伴历史接收清单 |

### 3.3 团队扩展性

- **新同事上手：** 不需要培训「每天去哪些网站找新闻」「怎么写推送文案」，直接说「采集今日全部内容」即可
- **多人协作：** SKILL.md 存在 Git 仓库中，团队可共同维护和优化指令逻辑
- **跨行业复用：** 修改数据源搜索策略和内容分类体系，其他渠道团队（如教育、政务）可以直接复用这套框架

## 四、局限性

> 以下是在搭建过程中识别到的真实局限，不回避不粉饰。

### 4.1 信息依赖网络公开来源

SaaS 行业资讯和飞书产品更新依赖 WebSearch 从互联网公开渠道获取，这意味着：
- **付费墙内容无法获取**（如 36kr 部分深度文章需要会员）
- **内部非公开信息无法覆盖**（如内部产品路线图、未公开的政策调整）
- **信息质量取决于搜索引擎返回结果的质量**

**缓解措施：** 飞书帮助中心的更新日志可以直接访问，是最可靠的来源。对于付费内容，可以继续手动录入补充。

### 4.2 推送依赖于群聊基础设施

目前推送通过飞书 IM 群聊实现，这意味着：
- 需要提前创建和维护四个分组的群聊
- 伙伴的进出需要手动管理群成员
- 如果某分组群聊中缺少 bot，推送会失败

**缓解措施：** 已在配置模板中提供群聊创建脚本，首次初始化一次性解决。后续可将群成员管理对接 lark-contact 实现自动化。

### 4.3 培训资料仍需人工提供

销售话术、客户案例、陪访总结等内部培训资料无法自动生成，需要渠道销售经理手动录入。这是业务特性决定的——高质量的内部培训内容依赖一线实战经验，AI 无法替代。

**缓解措施：** 这是「手动录入」而非「技能缺陷」。实际上，将培训资料与自动采集的行业资讯/产品更新分离，更能保证内容质量。

### 4.4 依赖 lark-cli 和 AI Agent 环境

- 需要安装 lark-cli 并完成飞书应用授权
- 需要在 AI Agent（如 Claude Code）环境下使用
- 初次配置约需 15-20 分钟（创建空间、群聊、授权）

**缓解措施：** 已在项目中提供完整的首次初始化指引和配置模板。

## 五、总结

这个 Skill 本质上解决了一个问题：**让渠道销售经理从「信息搬运工」回到「业务增长驱动者」的角色上。**

技术实现上，它不是一个复杂的自建系统，而是巧妙地利用了飞书 CLI 的开源能力和 AI Agent 的自然语言理解能力，用一份不到 2000 行的 SKILL.md 文件，把「搜集→加工→分发→归档」这个完整的内容运营链路自动化了。

成本极低（零额外服务器、零数据库、零前端开发），反馈极快（改 SKILL.md 即生效），且完全可定制（搜索策略、分组规则、推送格式都可以随时修改）。

下一步的方向是：在实际渠道业务中跑通完整周期，积累使用数据，迭代搜索策略和分组推送逻辑，并探索 FAQ 自动应答等扩展能力。
