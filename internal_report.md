# 飞书渠道伙伴内容推送 Skill — 内部汇报

> 汇报人：飞书渠道销售团队 | 2026.06.07
> 代码仓库：https://github.com/xmengLIU/feishu-channel-sales-content-skill

---

## 一、搭建思路

### 痛点

渠道销售每天有三个「隐形消耗」环节，合计耗时 **1 小时以上**：

1. **搜新闻** — 每天刷 36kr、牛透社、飞书更新日志，筛选值得同步给伙伴的内容（30-45 分钟）
2. **写文案** — 找到内容后要精简改写，不能直接转发原文链接（20-30 分钟）
3. **定向发** — 新伙伴要基础培训、核心伙伴要竞品分析、低效伙伴要销售技巧，但手动区分太麻烦，往往一刀切群发（15-20 分钟）

### 核心洞察

渠道伙伴赋能不能止于「发通知」。**信息时效性 = 渠道信任度，内容差异化 = 赋能精准度，可回溯 = 持续改进的基础。** 过去发完就完了，不知道发了多少次、哪些伙伴收到了什么——没有积累就无法优化。

### 解决思路

一份 SKILL.md 文件 + 飞书 CLI + AI Agent，把「搜集→加工→分发→归档」全链路自动化。**渠道经理从执行者变成决策者：决定推什么、推给谁，AI 完成剩下的全部操作。**

---

## 二、使用流程

每日只需 **3 步，约 10 分钟人工时间**：

```
09:30  Agent 自动搜索今日行业新闻 + 飞书更新 → 输出预览（含来源链接）
10:00  你花 3-5 分钟确认/删改 → 归档至知识库
16:00  生成推送文案 → 确认后一键发送
```

一句话概括交互原则：**AI 负责「找」和「写」，人负责「选」和「发」。** Agent 永不在未经确认的情况下推送。

---

## 三、应用价值

**提效：让日课不再消耗人。** 过去渠道经理每天到公司第一件事是刷新闻、找更新、改文案，不知不觉半个上午过去了——这是每天的「日课」，不做不行，但做了又觉得时间花得不值。现在 Skill 接管了「找」和「写」，日课从 80 分钟压到 10 分钟，释放出来的时间可以投入真正产生业务价值的事：跟进商机、陪访伙伴、做复盘。

**赋能：让推送不再是「群发」。** 一刀切群发本质是「用战术勤奋掩盖战略懒惰」——看似每天都发了，实际内容不匹配、伙伴不看、看了也没用。Skill 内置四级分组（全体/核心/新签/低效），每条内容自动匹配目标受众和差异化附言，让新伙伴学基础、核心伙伴看竞品、低效伙伴补技巧——推送从「完成动作」变成「产生效果」。

**沉淀：让经验不再随聊天记录消失。** 过去推送完就完了，发了什么、发给谁、谁看了什么——全部散落在聊天记录里，无法回溯。现在所有内容自动归档飞书知识库，按类别+日期+标签结构化存储。周度复盘时一键拉出本周推送清单，哪个分组覆盖不足一目了然。新同事入职不需要从头教，知识库里什么都有。

**杠杆：一份文件撬动全团队复用。** 整个 Skill 本质是一份 SKILL.md，存在 Git 仓库里。团队任何一个成员的优化（比如发现了一个更好的新闻源、调优了推送文案风格），改一行提交一次，全团队即时生效。不需要开发资源、不需要审批流程、不需要等排期。

---

## 四、局限性

1. **信息依赖公开网络。** 付费墙内容（如 36kr 深度文章）和飞书内部未公开信息无法自动获取，需手动补录。飞书帮助中心是主力来源，信息可靠。
2. **推送需要群聊基建。** 首次需创建四个分组的飞书群聊并拉入伙伴，后续群成员变动需手动维护。已在配置模板中提供自动化脚本。
3. **培训资料仍需人工录入。** 销售话术、客户案例、陪访总结等内部经验无法自动生成——这本质是业务特性，高质量内部内容依赖一线实战积累。
4. **依赖 lark-cli 和 AI Agent 环境。** 首次配置约 15-20 分钟（安装授权 + 创建空间群聊），已在项目中提供完整指引。

---

## 总结

**一句话：让渠道销售经理从「信息搬运工」回到「业务增长驱动者」。** 不是自建系统，一份 SKILL.md 就完成了内容运营全链路自动化——成本极低、反馈极快、完全可定制。
