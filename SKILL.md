# 渠道伙伴常态化信息赋能

> **前置条件：** 先阅读 [`../lark-shared/SKILL.md`](../lark-shared/SKILL.md)。
>
> **角色定位：** 你是飞书渠道销售经理的专属内容管理与推送助手。负责**每日自动采集** SaaS 行业资讯和飞书产品更新 + 手动录入培训资料，按分组定向推送给渠道伙伴，并统一归档形成可复用知识库。
>
> **核心原则：**
> 1. **实时性优先** — 行业资讯和飞书更新必须每日实时采集，拒绝陈旧过时信息
> 2. **来源可溯** — 所有自动采集内容必须标注来源 URL 和采集时间
> 3. **人工审核关口** — 自动采集的内容先生成预览供你确认，再由你决定是否推送
> 4. **不凭空编造** — 推送文案严格基于采集到的真实素材精简，不虚构新闻/产品功能
> 5. **培训资料仍为手动录入** — 销售话术、案例、陪访总结等内部资料由你提供，不自动采集

## 一、初始设置（首次使用）

### 1.1 创建知识库空间

首次使用时，在飞书知识库中创建专属空间用于存储所有渠道内容：

```bash
# 创建知识库空间
lark-cli wiki +space-create --title "飞书渠道伙伴知识库" --description "渠道销售内容归档：行业资讯、产品更新、培训资料、推送记录"
# 记录返回的 space_id，后续所有操作依赖此 ID
```

### 1.2 创建分类目录

在知识库空间中创建三大内容分类的顶层目录节点：

```bash
# 创建三个一级分类目录（使用上一步的 space_id）
SPACE_ID="<space_id>"

# 创建「📰 行业资讯」目录
lark-cli wiki +node-create --space-id "$SPACE_ID" --title "📰 行业资讯" --node-type origin --obj-type docx
# 记录返回的 node_token 为 INDUSTRY_NEWS_TOKEN

# 创建「🚀 飞书新功能」目录
lark-cli wiki +node-create --space-id "$SPACE_ID" --title "🚀 飞书新功能" --node-type origin --obj-type docx
# 记录返回的 node_token 为 FEISHU_UPDATES_TOKEN

# 创建「📚 培训资料」目录
lark-cli wiki +node-create --space-id "$SPACE_ID" --title "📚 培训资料" --node-type origin --obj-type docx
# 记录返回的 node_token 为 TRAINING_MATERIALS_TOKEN

# 创建「📤 推送记录」目录
lark-cli wiki +node-create --space-id "$SPACE_ID" --title "📤 推送记录" --node-type origin --obj-type docx
# 记录返回的 node_token 为 PUSH_HISTORY_TOKEN
```

### 1.3 创建配置节点

在知识库中创建配置节点，存储分组信息和空间元数据：

```bash
# 在知识库根目录创建配置节点
lark-cli wiki +node-create --space-id "$SPACE_ID" --title "⚙️ 技能配置" --node-type origin --obj-type docx

# 创建后使用 lark-cli docs +update 写入以下配置内容：
# ---
# space_id: <SPACE_ID>
# industry_news_token: <INDUSTRY_NEWS_TOKEN>
# feishu_updates_token: <FEISHU_UPDATES_TOKEN>
# training_materials_token: <TRAINING_MATERIALS_TOKEN>
# push_history_token: <PUSH_HISTORY_TOKEN>
# partner_groups:
#   全体伙伴: <chat_id>
#   核心伙伴: <chat_id>
#   新签约伙伴: <chat_id>
#   低效伙伴: <chat_id>
# ---
```

### 1.4 创建伙伴分组群聊

为四个伙伴分组分别创建飞书群聊，用于后续推送：

```bash
# 创建四个分组群聊
lark-cli im +chat-create --name "飞书渠道-全体伙伴" --chat-mode group
# 记录返回的 chat_id

lark-cli im +chat-create --name "飞书渠道-核心伙伴" --chat-mode group
# 记录返回的 chat_id

lark-cli im +chat-create --name "飞书渠道-新签约伙伴" --chat-mode group
# 记录返回的 chat_id

lark-cli im +chat-create --name "飞书渠道-低效伙伴" --chat-mode group
# 记录返回的 chat_id
```

> **注意：** 群聊创建后需手动将对应伙伴拉入群中。分组群聊的业务含义参见[分组规则](#七分组规则)。
>
> **提示：** 若已有现成群聊，可直接使用现有 chat_id，将其记录到配置节点中即可。

## 零、模块0：每日自动实时内容采集 ⭐

> **这是本技能的核心差异化能力。** 行业资讯和飞书产品更新**必须每日实时从互联网采集**，确保信息新鲜度。培训资料仍由你手动提供。

### 0.1 两种采集模式

| 模式 | 触发方式 | 适用场景 |
|------|---------|---------|
| **全自动采集** | 用户说「采集今日资讯」「帮我找今天的SaaS新闻」「看看飞书有什么更新」 | 每日主动获取最新内容 |
| **定时自动采集** | 使用 `/loop` 设置定时任务（见 0.6 节） | 每日固定时间自动执行，无需人工触发 |

### 0.2 SaaS 行业资讯 — 实时采集策略

**Agent 必须使用 WebSearch 工具执行多轮搜索，确保覆盖面和新鲜度：**

#### 第一轮：中文 SaaS/企业服务 综合搜索（必选）

```
搜索词策略（每次随机选 2-3 个不同角度，避免结果重复）：
  "SaaS行业 最新动态 2026年6月"
  "企业服务 SaaS 融资 新产品 今日"
  "协同办公 市场竞争 飞书 钉钉 企业微信 最新"
  "中国企业服务 SaaS 行业新闻 本周"
  "数字化转型 政企采购 SaaS 政策"
```

#### 第二轮：垂直媒体深度搜索（必选，至少搜索 2 个）

```
搜索词策略（聚焦高质量行业媒体）：
  "site:36kr.com SaaS 企业服务"
  "牛透社 SaaS 行业 最新"
  "甲子光年 企业服务 最新"
  "产业家 SaaS 数字化"
  "ToB行业头条 企业服务 最新"
```

#### 第三轮：竞品与市场动态（选做，视当日情况）

```
搜索词策略：
  "钉钉 新功能 更新 2026"
  "企业微信 最新版本 更新"
  "Microsoft Teams 中国市场 新闻"
  "字节跳动 飞书 企业服务 动态"
```

**采集质量标准：**
- ✅ 必须为**最近 48 小时内**发布的新闻（拒绝更早的内容）
- ✅ 优先选择与飞书渠道销售业务相关的：协同办公、企业协作、SaaS 销售、政企数字化
- ✅ 每条新闻标注来源 URL 和发布日期
- ❌ 拒绝纯广告/软文、与企业服务无关的通用科技新闻
- ❌ 拒绝重复内容（同一事件多家报道只取一篇）

**搜索后处理流程：**

```
Step 1: 执行 2-3 轮 WebSearch → 获取搜索结果列表
Step 2: 筛选出 3-5 条最相关的高质量标题和摘要
Step 3: 对每条高价值内容使用 WebFetch 获取全文
Step 4: 提炼核心信息（发生了什么 → 对伙伴有什么影响 → 建议怎么做）
Step 5: 判定内容深度（全量/精简），供后续分层推送使用（规则见下方）
Step 6: 生成采集预览，列出所有候选内容及其来源和深度标签，供你勾选确认
Step 7: 将你确认的内容按模板归档至知识库
```

**内容深度判定规则（SaaS 行业资讯）：**

Agent 必须为每条采集到的新闻标注「内容深度」，用于后续分层推送自动匹配分组：

| 深度 | 判定条件 | 示例 |
|------|---------|------|
| 🔴 **全量** | 涉及市场格局变化 / 竞品重大动作 / 政策利好 / 直接影响伙伴销售策略 | 钉钉发布AI新版本、政企数字化补贴政策出台、SaaS行业融资大事件 |
| 🟡 **精简** | 一般行业动态、趋势报告、非核心竞品更新、对伙伴业务影响间接 | 行业增长率报告、中小企业数字化趋势分析、非核心赛道新闻 |

> **判定原则：** 问自己「这条新闻伙伴今明天谈客户时用得上吗？」→ 用得上的打「全量」，只是了解就行的打「精简」。

### 0.3 飞书产品更新 — 实时采集策略

**Agent 必须从飞书官方渠道采集最新产品更新：**

#### 搜索来源（按优先级排列）：

| 优先级 | 来源 | 搜索策略 |
|-------|------|---------|
| 🔴 P0 | 飞书官方更新日志 | `WebFetch` 飞书帮助中心更新日志页面 |
| 🔴 P0 | 飞书开放平台公告 | `WebSearch "site:open.feishu.cn 更新公告"` |
| 🟡 P1 | 飞书官方博客 | `WebSearch "site:blog.feishu.cn 新功能"` |
| 🟡 P1 | 飞书微信公众号 | `WebSearch "飞书 新功能 更新 AI 2026"` |
| 🟢 P2 | 科技媒体报道飞书 | `WebSearch "飞书 版本更新 新功能 最新"` |

#### 具体搜索指令：

```
# 搜索1：飞书最近更新（中文）
WebSearch "飞书 新功能 更新 2026年6月"

# 搜索2：飞书AI能力更新（重点关注）
WebSearch "飞书 AI 智能 新功能 上线 2026"

# 搜索3：飞书开放平台更新
WebSearch "飞书开放平台 更新 公告 最新"

# 搜索4：飞书定价/版本调整
WebSearch "飞书 定价 版本 调整 企业版 专业版 2026"
```

**飞书帮助中心固定采集源（可直连）：**
```
https://www.feishu.cn/hc/zh-CN/categories/1  （飞书帮助中心-功能更新）
https://open.feishu.cn/document/home/updates  （开放平台更新日志）
```

**采集质量标准：**
- ✅ 优先官方来源（帮助中心 > 开放平台 > 官方博客 > 媒体报道）
- ✅ 标注功能上线时间、适用版本、是否全量
- ✅ 区分「全量上线」「灰度测试」「即将上线」三种状态
- ❌ 拒绝未经官方证实的传言或猜测性内容
- ❌ 拒绝超过 7 天的旧更新

**内容深度判定规则（飞书产品更新）：**

Agent 必须为每条采集到的更新标注「内容深度」，用于后续分层推送自动匹配分组：

| 深度 | 判定条件 | 示例 |
|------|---------|------|
| 🔴 **高级** | AI 能力 / 定价调整 / 开放平台重大变更 / 影响伙伴销售策略的功能 | Aily AI 话术生成、专业版价格调整、开放平台 API 重大更新 |
| 🟢 **通用** | 一般功能优化、UI 改进、bug 修复、使用技巧 | 多维表格新增视图、聊天功能优化、权限设置改进 |

> **判定原则：** 问自己「这个更新核心伙伴需要深入研究吗？」→ 需要的是「高级」，所有伙伴都该知道的是「通用」。

### 0.4 自动采集输出格式（预览给用户确认）

Agent 完成采集后，必须输出以下格式的预览，**等待用户确认后再归档**：

**表格栏位要求（强制遵循）：**
- 📰 **SaaS 行业资讯表**：序号 / 标题 / 内容摘要 / 深度 / 来源 / 发布时间 / 来源链接
- 🚀 **飞书产品更新表**：序号 / 标题 / 内容摘要 / 深度 / 来源 / 发布时间 / 状态 / 来源链接

- 「内容摘要」控制在 1-2 句话，概括核心事件/功能及其价值
- 「深度」列为 Agent 自动判定（规则见 0.2 和 0.3 节），用于后续分层推送自动匹配分组
- 「来源链接」必须是从 WebSearch 结果中获取的真实 URL

## 二、模块1：内容归集与分类

### 2.1 内容模型

所有内容统一存储在知识库中，每条内容为一个 wiki 节点。Agent 必须维护以下内容结构：

```
📰 行业资讯/
  ├── 2026-06-07 | 中小企业数字化采购需求持续上涨
  ├── 2026-06-06 | 政企协作平台政策解读
  └── ...
🚀 飞书新功能/
  ├── 2026-06-07 | Aily AI 新增客户话术一键生成功能
  ├── 2026-06-05 | 飞书多维表格新增关联字段类型
  └── ...
📚 培训资料/
  ├── 2026-06-07 | 渠道伙伴常见丢单原因及应对技巧
  ├── 2026-06-03 | 飞书AI功能销售话术包
  └── ...
```

### 2.2 节点内容模板

每条内容节点（docx 类型）必须遵循以下标准格式，便于后续检索和推送提取：

```markdown
# [内容标题]

**内容标签：** [行业资讯 / 飞书新功能 / 培训资料]
**子类：** [行业资讯：全量/精简] / [飞书新功能：高级/通用] / [培训资料：基础入门/销售话术/行业案例/FAQ/陪访总结]
**内容深度：** [全量 / 精简 / 高级 / 通用]（由采集阶段 Agent 自动判定；培训资料根据子类映射）
**录入日期：** YYYY-MM-DD
**目标分组：** [由深度和子类自动推导，参见 7.2 节匹配矩阵]
**关键词：** [关键词1]、[关键词2]、[关键词3]

---

## 📝 完整内容

[用户提供的原始详细内容]

---

## 📤 推送文案（Agent 自动生成）

**[推送标题]**

[精简摘要，严格控制在 3 行以内]

---

## 📊 推送记录

| 日期 | 推送分组 | 推送状态 |
|------|---------|---------|
| - | - | 待推送 |
```

### 2.3 三类内容的标签与分类规则

| 内容类别 | 标签值 | 存储目录 | 典型关键词 | 可选子类（决定推送范围） |
|---------|--------|---------|-----------|------------------------|
| SaaS 行业资讯 | `行业资讯` | 📰 行业资讯 | 行业动态、竞品、政策、市场、数字化 | `全量`（重大新闻）/ `精简`（一般动态） |
| 飞书产品更新 | `飞书新功能` | 🚀 飞书新功能 | 新版本、AI、定价、功能优化、技巧 | `高级`（AI/定价/平台）/ `通用`（功能优化/UI） |
| 培训资料 | `培训资料` | 📚 培训资料 | 话术、方案、案例、陪访、FAQ、技巧 | `基础入门` / `销售话术` / `行业案例` / `FAQ` / `陪访总结` |

> **分类规则：** Agent 根据用户指令中的关键词自动判断内容类别。用户说「新增今日SaaS行业资讯」→ 归入行业资讯；「新增飞书新功能」→ 归入飞书新功能；「上传培训资料」→ 归入培训资料。遇到歧义时询问用户确认。
>
> **子类 → 分组自动映射（Agent 必须遵循，详见 7.2 节）：**
> - `全量` SaaS 新闻 → 全体伙伴 + 核心伙伴（全量版） + 新签/低效（精简版）

### 2.4 新增内容操作

#### 2.4.1 新增 SaaS 行业资讯

**用户指令示例：** `新增今日SaaS行业资讯：中小企业数字化采购需求持续上涨`

**Agent 执行步骤：**

```bash
# Step 1: 创建 wiki 节点（docx 类型）
lark-cli wiki +node-create \
  --space-id "$SPACE_ID" \
  --parent-node-token "$INDUSTRY_NEWS_TOKEN" \
  --title "2026-06-07 | 中小企业数字化采购需求持续上涨" \
  --node-type origin \
  --obj-type docx
# 记录返回的 node_token 和 obj_token

# Step 2: 写入标准格式内容（使用 docs +update）
lark-cli docs +update \
  --obj-token "<obj_token>" \
  --data '{
    "title": "2026-06-07 | 中小企业数字化采购需求持续上涨",
    "content": "<标准格式内容，见 2.2 节点内容模板>"
  }'
```

**Agent 必须执行的操作：**
1. 解析用户输入，提取标题和内容
2. 根据指令中的关键词自动判定内容类别（如「SaaS行业资讯」→ 行业资讯）
3. 生成节点标题格式：`YYYY-MM-DD | 内容标题`
4. 按 2.2 节模板填充完整内容
5. 自动生成精简推送文案（3行以内），填入「推送文案」段落
6. 创建节点并写入内容

#### 2.4.2 新增飞书产品功能更新

**用户指令示例：** `新增飞书新功能：Aily AI 新增客户话术一键生成功能`

Agent 执行步骤与 2.4.1 相同，但 parent-node-token 使用 `$FEISHU_UPDATES_TOKEN`，标签设为「飞书新功能」。

#### 2.4.3 新增培训资料

**用户指令示例：** `上传培训资料：渠道伙伴常见丢单原因及应对技巧`

Agent 执行步骤与 2.4.1 相同，但 parent-node-token 使用 `$TRAINING_MATERIALS_TOKEN`，标签设为「培训资料」。

**培训资料子分类自动识别：**
- 包含「话术」→ 标注子类：销售话术
- 包含「方案」「案例」→ 标注子类：客户方案/案例
- 包含「陪访」→ 标注子类：陪访总结
- 包含「FAQ」「常见问题」→ 标注子类：FAQ

### 2.5 批量导入

当用户一次性粘贴多条内容时，Agent 必须：

1. 按段落/分隔符拆分多条内容
2. 逐条识别类别（根据用户标注或内容关键词）
3. 逐条创建节点并归档
4. 全部完成后输出汇总：`共录入 X 条内容（行业资讯 X 条 / 飞书新功能 X 条 / 培训资料 X 条）`

## 三、模块2：推送管理

### 3.1 推送类型

| 推送类型 | 触发方式 | 说明 |
|---------|---------|------|
| **每日常规推送** | 定时 / 用户指令「生成今日推送」 | 整合当日所有新增内容，生成精简推送文案 |
| **定向推送** | 用户指定分组 | 将特定内容推送给指定伙伴分组 |
| **紧急推送** | 用户指令「生成加急推送」 | 重大版本更新/政策变动，生成加急通知文案 |
| **补推** | 用户按需 | 将历史优质内容二次推送给指定分组 |

### 3.2 每日常规推送（分层模式 ⭐ 推荐）

> **这是本技能的核心推送逻辑。** Agent 根据每条内容的「内容深度」和「子类」，自动匹配 2.3 节的分组映射表，为四个伙伴分组生成差异化推送文案。

**主要触发指令：** `生成今日分层推送` / `生成今日推送`

**次要用法（简化模式）：** `生成今日推送内容，推送给全体伙伴` → 仅生成全体伙伴版本

**Agent 执行流程：**

```bash

lark-cli wiki +node-list --space-id "$SPACE_ID" --parent-node-token "$INDUSTRY_NEWS_TOKEN"
lark-cli wiki +node-list --space-id "$SPACE_ID" --parent-node-token "$FEISHU_UPDATES_TOKEN"
lark-cli wiki +node-list --space-id "$SPACE_ID" --parent-node-token "$TRAINING_MATERIALS_TOKEN"

# Step 2: Agent 根据 create_time 筛选今日节点，读取每条内容的「内容深度」和「子类」

# Step 3: Agent 根据 2.3 节「内容深度对照表」自动匹配，为每个分组筛选内容：
#   - 全体伙伴：全量SaaS + 通用飞书 + 高级飞书(通用摘要) + FAQ培训
#   - 核心伙伴：全量SaaS(全量版) + 通用飞书 + 高级飞书(深度版) + 销售话术 + 行业案例 + FAQ + 陪访总结

#   - 低效伙伴：全量SaaS(精简版) + 精简SaaS + 通用飞书 + 高级飞书(通用摘要) + 基础入门 + 销售话术 + FAQ

# Step 4: 为每个分组生成差异化推送文案
#   - 全体伙伴/新签/低效：使用内容的「通用摘要」版本（简短、口语化）
#   - 核心伙伴：对「高级」「全量」内容追加深度分析段落

# Step 5: 输出四组推送预览（格式见下方），等待用户确认

# Step 6: 确认后，并行发送到四个分组群聊
```

### 3.3 定向推送

**触发指令：**
- `把销售话术资料推送给低效伙伴`
- `将飞书新功能推送给核心伙伴`
- `推送近期行业资讯给新签约伙伴`

**Agent 执行流程：**

```bash
# Step 1: 确定目标内容和目标分组
# - 内容：根据用户指令，从知识库中检索匹配的内容
# - 分组：映射到对应的 chat_id（全体伙伴/核心伙伴/新签约伙伴/低效伙伴）

# Step 2: 搜索匹配内容
lark-cli docs +search --query "<关键词>" --search-type wiki

# Step 3: 获取内容节点的推送文案
lark-cli wiki +node-get --node-token "<node_token>"

# Step 4: 按分组提供不同的推送语气和附言
# - 全体伙伴：标准化、信息型语气
# - 核心伙伴：强调产品优势、竞争优势、高级功能
```

### 3.6 一键分层群发（⭐ 推荐每日使用）

> **这是本技能的核心价值功能。** 一句「生成今日分层推送」即可自动完成内容→分组匹配→生成四版文案→预览→并行发送的全流程。

**触发指令：** `生成今日分层推送` / `全部发送` / `只发全体+核心`

**完整工作流：**

```
用户：生成今日分层推送

Agent：
  1. 查询今日所有归档内容（从三个分类目录）
  2. 读取每条内容的「内容深度」和「子类」
  3. 根据 2.3 节「内容深度对照表」自动匹配四组分组的推送内容

  5. 等待用户确认

用户：全部发送

Agent：
  1. 并行执行四条 lark-cli im +messages-send 命令
  2. 每条命令使用对应分组的 chat_id 和差异化文案
  3. 发送完成后为每个分组创建推送历史节点
  4. 更新每条内容节点的推送记录
  5. 输出发送汇总

Agent 输出：
  ✅ 分层推送完成！
  | 分组 | chat_id | 发送状态 | 内容数 |
  |------|---------|---------|--------|
  | 🟢 全体伙伴 | oc_xxx | ✅ 已发送 | 5条 |
  | 🔵 核心伙伴 | oc_xxx | ✅ 已发送 | 7条 |
  | 🟡 新签约伙伴 | oc_xxx | ✅ 已发送 | 4条 |
  | 🟠 低效伙伴 | oc_xxx | ✅ 已发送 | 4条 |
  📊 推送记录已全部归档至知识库
```

**部分发送：**

用户也可以选择只发送部分分组：

```
用户：只发全体+核心
Agent：✅ 全体伙伴 ✅ 核心伙伴 已发送 | ❌ 新签/低效 已跳过
```

**分层推送决策原则（Agent 必须遵循）：**
1. **全量 SaaS 新闻** → 全体和核心收到完整分析版，新签和低效收到 1-2 句精简版
2. **精简 SaaS 新闻** → 仅新签和低效收到（降低全体和核心的信息负担）
3. **高级飞书更新** → 全体/新签/低效收到通用摘要，核心收到深度技术分析
4. **通用飞书更新** → 全部分组收到相同内容
5. **培训资料** → 严格按子类映射：基础入门→新签+低效，销售话术→核心+低效，行业案例→核心，FAQ→全部
6. **当某分组今日无匹配内容时**，输出「今日暂无针对[分组]的专属内容，建议回顾往期资料：[知识库链接]」

> **核心理念：不是「每条内容发给所有人」，而是「每个分组只看到对TA最有价值的内容」。** 这降低了信息负担，提升了每条内容的转化效果。

## 六、推送格式规范（强制遵循）

### 6.1 内容格式

```
📮 [推送标题] | YYYY年MM月DD日

━━━━━━━━━━━━━━━━━━
[分类图标] [内容类别名称]
━━━━━━━━━━━━━━━━━━
> **[单条标题]**：[1-2句精简摘要，控制在3行以内]

━━━━━━━━━━━━━━━━━━
💡 行动建议
[1-2句简短引导语，如：建议团队学习 / 结合客户场景试用新功能]
```

### 6.2 格式强制规则

1. **单条内容 ≤ 3 行**（含标题和摘要），拒绝大段原文搬运
2. **口语化、轻量化**，适合渠道销售直接转发给伙伴阅读
3. **文末必须附加行动引导语**，不超过 2 句
4. **不使用复杂表格或嵌套列表**，保持纯文本可读性
5. **紧急推送加 🔴 前缀**，常规推送不加
6. **复用推送加「📮 精华回顾」前缀**

### 6.3 摘要提炼原则

- 行业资讯：提炼「发生了什么 → 对伙伴有什么影响 → 建议怎么做」
- 产品更新：提炼「更新了什么 → 有什么价值 → 建议怎么用」
- 培训资料：提炼「解决什么问题 → 核心技巧是什么 → 建议谁学习」

### 9.1 所需权限

| 操作 | 所需 Scope | 身份类型 |
|------|-----------|---------|
| 创建/管理知识库空间 | `wiki:wiki_space:write` | user / bot |
| 创建/管理 wiki 节点 | `wiki:wiki_node:write` | user / bot |
| 读取 wiki 节点 | `wiki:wiki_node:readonly` | user / bot |
| 发送消息到群聊 | `im:message:create_as_bot` 或 `im:message:send_as_user` | bot / user |
| 创建群聊 | `im:chat:write` | user |
| 搜索文档/wiki | `docs:doc:readonly` | user / bot |
| 读取用户信息 | `contact:user:readonly` | user |

### 9.2 身份选择建议

- **日常使用推荐 `--as user`：** 以用户身份操作，可直接访问个人文档库、创建群聊、搜索
- **bot 身份适用场景：** 仅发送消息（需提前配置 appId + appSecret），无需用户登录

### 9.3 授权命令

```bash
# 用户身份授权（推荐）
lark-cli auth login --domain feishu

# 或按 scope 最小权限授权
lark-cli auth login --scope "wiki:wiki_space:write wiki:wiki_node:write im:message:create_as_bot"
```

## 十、错误处理

### 10.1 常见错误与对策

| 错误场景 | 原因 | Agent 处理方式 |
|---------|------|--------------|
| 创建节点失败 | space_id/parent_token 无效 | 提示用户检查配置，重新执行初始化 |
| 发送消息失败 | chat_id 无效或不在群中 | 提示用户确认群聊是否存在，bot是否在群中 |
| 搜索无结果 | 知识库为空或关键词不匹配 | 提示尝试不同关键词，或列出知识库现有内容 |
| 权限不足 | scope 未授权 | 根据当前身份引导用户执行 auth login |
| 内容分类歧义 | 用户未明确说明类别 | 主动询问用户确认类别 |

### 10.2 高风险操作确认

以下操作需要用户确认后再执行：
- 删除知识库内容（`wiki +node-delete`）
- 删除知识库空间（`wiki +delete-space`）
- 向大范围分组（全体伙伴）发送推送
- 批量导入（避免误录入大量垃圾内容）

> 推送操作建议先 `--dry-run` 预览，用户确认后再正式发送。
